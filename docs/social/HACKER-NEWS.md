Hi HN,

I'm building AI4Change — an open-source initiative that connects communities facing real-world problems with global contributors who use AI as their tool to build solutions.

**The inspiration**: Playing for Change (https://playingforchange.com) films musicians on street corners across the globe, each playing their part of the same song independently. Layered together, the result is something no single studio could produce. We apply the same model to AI-powered solutions.

**How it works**:

1. Anyone submits a real-world problem (via GitHub Issues)
2. Community discusses and signals which problems matter most
3. Top problems become pilot projects ("songs")
4. Contributors worldwide collaborate — each using AI tools as their instrument
5. Everything ships open source (MIT)

**What makes this different from $EXISTING_THING**:

- vs. hackathons: We build continuously, not for 48 hours
- vs. nonprofits: You need a GitHub account, not a funding cycle
- vs. AI ethics orgs: We ship code, not papers
- vs. typical OSS: We start with the problem, not the solution

**Problem areas we're collecting**: agriculture (crop disease prediction), education (offline literacy tools), health (disease outbreak tracking), environment (air quality monitoring), disaster response (early warning systems).

**Current state**: Phase 1 — collecting seed problems, building the community foundation. No code yet for solutions, which is intentional. The community chooses what we build. The tech stack for each project will be driven by the problem requirements.

**Technical details**: GitHub-native workflow. Problems are Issues, community discussion lives in GitHub Discussions, the website is static on GitHub Pages. Each pilot project gets its own repo. We're particularly interested in edge deployment, offline capability, and low-resource hardware — because the target users often don't have reliable internet or expensive devices.

GitHub org: https://github.com/ai4change-org
Manifesto: https://github.com/ai4change-org/.github/blob/main/docs/MANIFESTO.md
Roadmap: https://github.com/ai4change-org/.github/blob/main/docs/ROADMAP.md

Would love feedback on:
- The problem-first model — does it hold up at scale?
- Blind spots in the governance/community approach?
- Similar efforts we should learn from or collaborate with?

## HN-Specific Tips

### Before Posting

- **Timing**: Post between 8-10am EST on a weekday (Tuesday-Thursday preferred)
- **Don't ask for upvotes** — HN actively penalizes vote rings
- **Have the website ready** — HN will click through immediately and judge the site
- **Check for duplicates** — search HN for "AI for good," "open source social impact," etc.

### Tone

- **Technical and honest** — HN respects intellectual honesty over marketing
- **Acknowledge what you don't know** — "we're figuring this out" beats "we've solved it"
- **Be specific** — concrete examples over abstract vision
- **Self-deprecating > self-promoting** — lead with what's hard, not what's great

### Handling Common HN Responses

| Response Type | How to Reply |
|:-------------|:------------|
| "This is just another feel-good project that won't ship" | Acknowledge the risk honestly. Point to the roadmap and concrete milestones. "You might be right — and that's why we're structured as continuous open source, not a hackathon. If we don't ship, we've failed publicly." |
| "Why not just contribute to existing projects?" | "Great question. We're not building a platform — we're building a community-driven pipeline from problem to solution. If an existing project solves a community-submitted problem, we'll contribute there instead of reinventing." |
| "The Playing for Change analogy doesn't hold" | "Fair pushback. The analogy isn't perfect — music has a natural mixdown point that code doesn't. What we're borrowing is the principle: real people, distributed globally, each contributing their part, composing into something none could build alone." |
| "How is this different from [X]?" | Be genuinely curious. Ask what X is, compare honestly, and suggest collaboration if relevant. |
| "This needs a business model" | "We're exploring grants and sponsorships, but the core constraint is: solutions must be open source and free. We'd rather stay small and open than scale and close." |

### Follow-Up Engagement

- **Reply to every substantive comment** within 2 hours
- **Don't argue** — if someone disagrees, ask what they'd do differently
- **Share technical details** when asked — HN loves depth
- **Thank constructive criticism** genuinely
- **If it makes the front page**: be prepared for 5,000+ visitors to the GitHub org and website within 24 hours

### What NOT to Do on HN

- Don't use emoji in the post
- Don't use marketing language ("revolutionary," "disruptive," "game-changing")
- Don't pitch — describe
- Don't make claims you can't back up with links
- Don't respond defensively to criticism
- Don't post in the middle of the night (US time)
